# 开发规范

本文件同时保存项目级工程基线和通用开发规则。`gj-codebase-map` 可以根据现有代码起草
“待确认”内容，但只有 Dev Lead 确认后才成为后续开发和审阅必须遵守的规范。技术栈、
架构边界或开发约定变化时，在同一个 MR 原地更新本文件；重大取舍另建 ADR。

## 项目技术基线

- 主要语言、运行时及允许版本：待确认。
- 前端框架、组件库、状态管理和构建工具：不适用或待确认。
- 后端框架、依赖管理和构建工具：不适用或待确认。
- 本地启动、构建和格式化命令：待确认。
- 配置文件、环境变量和本地秘密边界：待确认；秘密值不得写入本文。

## 架构和目录边界

- 入口、分层、依赖方向和禁止跨越的边界：待确认。
- 前端页面、组件、状态、请求和公共能力目录：不适用或待确认。
- 后端接口、应用、领域、数据访问和基础设施目录：不适用或待确认。
- 模块路径和负责人以 `docs/context/module-map.md`、`docs/modules/*.md` 和
  `.gj/context.yml` 为准。
- 长期架构取舍写入 `docs/technical/decisions/`，不能根据现有代码习惯自动推断为决策。

## 工程约定

### 通用

- 命名、目录、代码格式、静态检查和生成代码规则：待确认。
- 错误处理、日志、配置、时间、金额和敏感数据规则：待确认。
- 新增依赖、外部集成或跨模块调用的评审要求：待确认。

### 前端

- 组件边界、状态管理、请求封装、路由、权限、表单和异常/空态约定：不适用或待确认。

### 后端

- 分层、DTO、事务、幂等、并发、权限、错误码、日志和数据访问约定：不适用或待确认。

测试框架、目录、命令、覆盖和 CI 要求统一见 `07-test-standard.md`；API、数据库、安全和
环境专项要求分别见 `03-api-standard.md`、`04-database-standard.md`、
`08-security-standard.md` 和 `10-environment-standard.md`，不要在本文件复制冲突规则。

## 交付规则

- Standard 和 Hotfix 变更必须关联 GitLab Issue。低风险 Fast MR 在描述中写清动机和
  范围后可以不建 Issue。
- 实现前确认唯一流程标签，并在 MR 中沿用：`flow::fast`、`flow::standard` 或
  `flow::hotfix`。
- 变更范围不得超出已确认的需求和方案。
- 行为变化时新增或更新测试。
- 在 MR 描述中记录 AI 使用范围。
- 变更业务规则时，在同一 MR 更新对应的 `docs/modules/*.md`。SmallChange MR 也适用：
  Fast 可以省略计划文档，但不能省略已变化的长期事实。
- 语言、框架、运行时、构建工具、架构边界或工程约定变化时，同一 MR 更新本文件；
  模块边界变化时同时更新 `module-map.md`、模块文档和 `.gj/context.yml`。

## 执行点

- 唯一 flow、Standard/Hotfix Issue 关联、高风险路径和新增 secret:CI
  `policy_check` 硬检查。
- MR 章节完整性:`policy_check` 只告警，由评审员判断是否足够。
- 文档决策和持久事实回写:对应 Skill 执行，评审员或发布责任人人工确认。
  `context_freshness_check` 仅供按需审计，不进入默认 CI。
