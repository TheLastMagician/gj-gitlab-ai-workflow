# 可持续文档与 AI 上下文治理

## 目标

文档既要支撑后续迭代，也不能让 AI 每次加载整个历史。核心方法是：当前事实与历史
证据分离、按模块和功能拆分、通过 `.gj/context.yml` 白名单渐进加载，并用按需
审计工具报告常驻上下文预算。

## 五层文档模型

| 层级 | 内容 | 默认加载方式 | 生命周期 |
| --- | --- | --- | --- |
| 工作项 | 当前 Issue、MR、讨论、验收与 Pipeline | 每个任务首先读取 | GitLab 保存过程历史 |
| 全局当前事实 | current state、module map、glossary | `always_load`，最多 3 个文件/24000 字符 | 原地覆盖，只写当前为真 |
| 模块当前事实 | `docs/modules/<module>.md` 和有效决策 | changed paths 命中模块后加载，最多 5 个文件/40000 字符 | 原地覆盖，一个有界模块一组 |
| 功能交付文档 | PRD、设计、方案、测试、发布说明 | 仅由 Issue/MR 或模块文档链接时加载 | 一个功能或稳定决策一个文件 |
| 迭代历史 | `docs/iterations/`、旧 release evidence | 默认不读；追溯时指定文件读取 | 关闭后冻结 |

每个模块只在 `recent_iteration_summaries` 保留最新 1 轮摘要。旧摘要仍在 Git 历史和
迭代目录中，但不会进入日常 AI 上下文。

## 为什么不能写成一个大文档

单一大文档会把无关模块、过时决策和历史过程一起加载，造成上下文浪费，也让多人修改
产生冲突。出现以下任一情况就应该拆分：

- 内容属于不同业务模块或 Owner；
- 不同部分有独立的评审、发布或失效周期；
- 一个任务只需要其中一部分，却不得不加载整份文件；
- 文件同时混合当前规则、讨论过程和历史变更。

拆分按稳定边界进行，不按每次迭代复制一套长效文档。迭代结束时，把确认后的当前事实
覆盖回长效文档，过程证据留在 Issue/MR 和冻结的迭代目录。

## AI 读取算法

1. 读取当前 Issue/MR，明确目标、flow 和预计 changed paths。
2. 读取 `.gj/context.yml`，只加载 `always_load`。
3. 用 changed paths 匹配一个或少数相关模块，只加载这些模块的 `docs`、有效决策和
   最新一轮摘要。
4. 仅在工作项明确链接时读取 PRD、设计、方案、测试或发布文档。
5. 不遍历 `docs/iterations/`；只有用户要求追溯某项决策时读取指定历史文件。

如果上下文仍然过大，先缩小模块或任务范围，而不是扩大常驻白名单。

## AI 写入算法

1. 判断本次是否产生持久事实。
2. 选择最小目标：产品规则、模块规则、技术决策、测试证据、发布证据或全局当前状态。
3. 结论已确认且可写时，在当前改动/MR 直接更新目标文件。
4. 结论未确认时只输出草稿和问题，不写成长期事实。
5. `gj-close-loop` 覆盖 current state、更新模块文档、修剪 context index，并只登记最新
   `ai-context-summary.md`。

## Skill 职责

| Skill | 文档职责 |
| --- | --- |
| `gj-workflow-next` | 识别阶段与文档缺口，不批量读取或修改文档 |
| `gj-plan-change` | 按需更新 PRD、设计、技术方案和测试计划 |
| `gj-develop-change` | 在同一 MR 更新模块规则、测试和实现相关说明 |
| `gj-mr-review` | 检查应更新的文档是否存在且与代码一致 |
| `gj-release-readiness` | 更新测试报告、发布说明、验证和回滚证据 |
| `gj-close-loop` | 更新长效上下文、迭代摘要和 context index |

人负责确认需求、方案和最终事实；不需要逐个记住文档路径或另开文档维护任务。

## 按需审计

`context_freshness_check.py` 默认以告警方式检查：

- 常驻文件和模块文档是否超出数量或字符预算；
- 常驻上下文是否错误引用迭代历史；
- context index 是否引用不存在的文件；
- 每模块最近摘要是否超过上限，最新摘要是否排在第一；
- 发布关闭时 current state 和迭代摘要是否完成回写。

该脚本不进入默认 CI，也不阻塞 MR。需要进行文档治理专项审计时才运行
`python scripts/context_freshness_check.py --strict`。文档事实是否正确，仍由对应角色在
计划、Review、发布和收尾决策门确认。
