# Skill 坏案例集(eval-cases)规约

每个 skill 目录下可以有一个 `eval-cases/` 目录,存放该 skill 曾经
处理错误(漏报、误报、跑偏)的真实案例,作为改提示词后的回归评估集。

## 为什么

提示词就是代码:评审员漏报了一个该抓的问题,处理方式和修 bug 一样——
开 issue、改提示词、用漏掉的案例回归验证。不积累案例集,skill 质量
停在第一版,团队用两次觉得不准就再也不看它的产出了。

## 目录结构

```text
skills/<skill-name>/eval-cases/
  001-<short-slug>/
    case.md        # 必填:案例说明(见下)
    input/         # 可选:当时的输入(diff、issue 描述、日志摘录)
```

## case.md 格式

```markdown
# <一句话:什么被漏掉/搞错了>

## 来源

<GitLab issue/MR 链接,发生日期>

## 当时的输入

<或指向 input/ 目录>

## skill 当时的产出(错误)

## 期望的产出

## 修复

<改了提示词的哪一段,对应 MR>
```

## 使用方式

- 改任何 skill 的提示词前,先跑一遍该 skill 的 eval-cases,
  确认已有案例仍然能被正确处理(人工跑即可,不要求自动化)。
- 新漏报/误报出现时,先补案例,再改提示词——和"先写失败测试再修复"
  同一纪律。
- 案例含敏感信息时脱敏后入库(遵循 08-security-standard)。
